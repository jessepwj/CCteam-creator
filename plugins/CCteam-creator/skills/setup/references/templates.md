This file is generated in the **project working directory** (not inside `.plans/`). Claude Code always loads it into the main session context, so team-lead never loses operational knowledge after context compression.

## Task Dispatch Protocol

### Large Tasks (features, new modules) -- IMPORTANT

When assigning a large task to any agent, your message MUST include:
1. **Scope & goals**: What to build, acceptance criteria
2. **Document reminder**: "Create a task folder `<prefix>-<task-name>/` with task_plan.md + findings.md + progress.md, and add an index entry to your root findings.md"
3. **Dependencies**: Which research/tasks they depend on, key file paths and line numbers
4. **Review expectation**: Whether code review is needed after completion

Example:
```
SendMessage(to: "backend-dev", message:
  "New task: implement auth module.
   Scope: JWT login + refresh token + auth middleware.
   Depends on: researcher's findings at .plans/<project>/researcher/research-auth/findings.md
   Please create task-auth/ folder and update your root findings.md index.
   This is a large feature -- request reviewer after completion.")
```

Task folder prefixes by role:
- backend-dev / frontend-dev: `task-<name>/`
- researcher: `research-<topic>/`
- e2e-tester: `test-<scope>/`
- reviewer: `review-<target>/`

### Small Tasks (bug fixes, config changes)

Direct message with the change description. No task folder or review needed.
```
SendMessage(to: "frontend-dev", message: "Fix XSS in login form, see src/auth/login.tsx:42")
```

## Communication Quick Reference

| Action | Command |
|--------|---------|
| Assign task to one agent | `SendMessage(to: "<name>", message: "...")` |
| Broadcast to all (use sparingly) | `SendMessage(to: "*", message: "...")` |
| Dev requests review | Dev contacts reviewer directly (not through team-lead) |

## Status Check

| What to Check | How |
|---------------|-----|
| Quick scan | Read each agent's `progress.md` in parallel |
| Deep dive | Read agent's `findings.md` (index) then specific task folder |
| Direction check | Read `.plans/<project>/task_plan.md` |

Reading order: **progress** (where are we) -> **findings** (what happened) -> **task_plan** (what's the goal)

## Key Protocols

| Protocol | Trigger | Action |
|----------|---------|--------|
| 3-Strike escalation | Agent reports 3 failures | Read their progress.md, give new direction or reassign |
| Code review | Large feature/module complete | Dev writes change summary in findings.md, sends to reviewer |
| Phase advance | Phase complete | Research done: read findings, update main plan. Dev done: wait for reviewer [OK]/[WARN] |
| Context overflow | Agent reports long context | Progress saved in files, resume agent or spawn successor |

## File Structure

```
.plans/<project>/
  task_plan.md          -- Main plan (team-lead maintains)
  findings.md           -- Team-level findings
  progress.md           -- Work log
  <agent-name>/         -- Per-agent directory
    task_plan.md        -- Agent task list
    findings.md         -- INDEX linking to task folders
    progress.md         -- Agent work log
    <prefix>-<task>/    -- Task folder (per assigned task)
      task_plan.md / findings.md / progress.md
```

## Task Folder Templates

All roles use task folders when assigned distinct tasks. The folder prefix varies by role:

| Role | Prefix | Example |
|------|--------|---------|
| backend-dev / frontend-dev | `task-` | `task-auth/`, `task-payments/` |
| researcher | `research-` | `research-tech-stack/`, `research-auth-options/` |
| e2e-tester | `test-` | `test-auth-flow/`, `test-checkout/` |
| reviewer | `review-` | `review-auth-module/`, `review-payments/` |
| cleaner | (uses root files) | — |
